# Core Concepts

## Locks (Decisions)

A **lock** is an immutable record of a product decision. Every lock captures:

| Field | Description |
|-------|-------------|
| **Message** | The decision statement (e.g., "Use notional value instead of margin") |
| **Short ID** | Human-readable identifier like `l-a7f3e2` |
| **Product** | Which product this applies to |
| **Feature** | Which feature area within the product |
| **Scope** | Impact level: `minor`, `major`, or `architectural` |
| **Author** | Who made the decision (human or agent) |
| **Source** | Where it was recorded (Slack, CLI, MCP, API) |
| **Context** | Surrounding discussion, thread snippet, participants |
| **Tags** | Free-form labels for categorization |
| **Links** | References to Jira tickets, GitHub PRs, Figma files, etc. |

### Immutability

Locks cannot be edited after creation. This is by design — decisions are historical records. If a decision changes, you either:

- **Revert** it: marks the original as reverted and creates a new record explaining why
- **Supersede** it: a new decision automatically replaces the old one (detected by Lock's conflict engine)

### Statuses

| Status | Meaning |
|--------|---------|
| `active` | Current, in-effect decision |
| `superseded` | Replaced by a newer decision |
| `reverted` | Explicitly rolled back |
| `proposed` | Suggested but not yet confirmed |
| `auto` | Automatically generated by an agent |

---

## Products

A **product** is the top-level organizational unit. It maps to whatever your team considers a distinct product or system.

Examples: "Trading Platform", "Risk Engine", "Mobile App", "Internal Tools"

Products are identified by a URL-safe **slug** (e.g., `trading`, `risk-engine`). They are auto-created on first reference — no admin setup required.

---

## Features

A **feature** is a scoped area within a product. It represents a workstream, epic, or functional area.

Examples: "Margin Rework", "Position Display", "User Onboarding", "API v2"

Features belong to exactly one product and are also identified by a slug. Like products, they are auto-created on first reference.

### Feature + Channel Mapping (Slack)

In Slack, you map a channel to a product and feature using `@lock init`. After that, every `@lock` command in that channel automatically targets that product and feature — no flags needed.

---

## Scopes

Every decision has a **scope** that indicates its impact:

| Scope | Emoji | When to Use |
|-------|-------|-------------|
| `minor` | :small_blue_diamond: | Day-to-day implementation choices. "Use a Map instead of an Object here." |
| `major` | :large_orange_diamond: | Significant product or technical decisions. "Switch from REST to GraphQL for the public API." |
| `architectural` | :rotating_light: | Foundational decisions that affect the entire system. "Migrate from monolith to microservices." |

The default scope is `minor`. Use `--scope major` or `--scope architectural` to override.

Scope affects how prominently a decision appears in recaps and how aggressively conflict detection surfaces it.

## Conflict Detection

When a new lock is committed, Lock automatically checks it against all active decisions in the same product. This works across features — a decision in "Margin Rework" is checked against decisions in "Position Display" and "Risk Engine" if they're all under the same product.

### How It Works

1. **Embedding generation**: Lock converts the decision text into a 1536-dimensional vector using OpenAI's `text-embedding-3-small` model
2. **Similarity search**: Lock queries PostgreSQL (pgvector) for the top 5 most similar active decisions in the same product
3. **LLM classification**: For each similar decision (above a 0.75 similarity threshold), Lock uses Claude to classify the relationship

### Relationship Types

| Relationship | Meaning |
|-------------|---------|
| `no_relation` | These decisions are about different things |
| `related` | Same area, but no contradiction |
| `potential_conflict` | These decisions may contradict each other |
| `supersession` | The new decision replaces the old one |

### Supersession

When Lock detects that a new decision replaces an older one, it automatically:

1. Sets the old decision's status to `superseded`
2. Links the two decisions (old.superseded_by = new, new.supersedes = old)
3. Reports the supersession in the response

This creates a **lineage chain** — you can trace how a decision evolved over time.

---

## Lineage

Every lock can be part of a chain:

```
Original Decision (l-a1b2c3) [superseded]
    |
    v  superseded by
Decision v2 (l-d4e5f6) [superseded]
    |
    v  superseded by
Decision v3 (l-g7h8i9) [active]
    |
    v  reverted by
Revert (l-j0k1l2) [active]
```

Use `lock show <id>` (CLI) or `lock_get_lineage` (MCP) to view the full chain for any decision.

## Cross-Surface Awareness

Lock is designed so that decisions made on one surface are visible everywhere:

- A decision made in **Slack** appears in `lock log` (CLI) and `lock_query` (MCP)
- A decision made via **CLI** posts a notification to the relevant Slack channel
- A decision made by an **AI agent** (MCP) also notifies Slack

This ensures the team always has visibility, regardless of where a decision originated.

### Slack as the Awareness Layer

Even when decisions come from the CLI or MCP, they're surfaced in Slack. Slack is where the team "sees" what's happening. If a feature has a linked Slack channel, Lock will post a notification there:

```
New lock from cli (l-a7f3e2)
  "Cache user sessions in Redis"
  Author: philippe via cli
```

## Authentication

Lock supports two authentication methods:

### API Keys

For CLI and MCP usage. API keys follow the format `lk_` + 32 hex characters. They're stored as SHA-256 hashes in the database — the raw key is only shown once at creation time.

Each key has:
- A human-readable name (e.g., "philippe's CLI")
- A prefix for identification (e.g., `lk_a7f3e2...`)
- Scopes (`read`, `write`)

### Internal Authentication

For the Slack bot, which communicates with the core API using a shared internal secret. This avoids needing per-user API keys for Slack interactions.

---

## Tags

Tags are free-form labels you can attach to any decision for categorization:

```
@lock Use Redis for session caching --tag performance --tag infrastructure
```

Tags are useful for filtering decisions later:

```
lock log --tag performance
```

---

## External Links

Locks can reference external resources:

| Link Type | Example |
|-----------|---------|
| `jira` | `TRADE-442` |
| `github` | `https://github.com/org/repo/pull/123` |
| `figma` | `https://figma.com/file/...` |
| `linear` | `ENG-456` |
| `notion` | `https://notion.so/page/...` |
| `other` | Any URL or reference |

Add links at commit time with `--ticket` (Jira shorthand) or after the fact with `lock link`.
